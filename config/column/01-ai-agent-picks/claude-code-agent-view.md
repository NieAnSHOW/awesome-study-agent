![](https://fastly.jsdelivr.net/gh/bucketio/img10@main/2026/05/12/1778549193822-f48d26a0-1977-4184-b674-ce009d459e42.png)
> 当你有 5 个 Agent 同时在跑，bug 修复、PR 评审、测试流水线、代码审查、文档更新各占一个终端标签页——你的大脑还能记住每个会话停在哪了吗？Claude Code 最近推出的 Agent View，就是来解决这个问题的。


## 问题：过载的心理账本

用 Claude Code 的独立开发者都有过这种经历——

你在终端里开了四五个标签页（或者 tmux 网格铺满了小窗），每个窗口里跑着一个 Agent 会话：一个在做重构，一个在跑测试，一个在审查 PR，还有一个在写文档。你从左到右挨个看，试图记住每个会话的上下文：「这个跑完了吗？」「那个是在等我输入吗？」「这个报错是什么时候的事？」


Anthropic 在发布博客里用了个很精准的词——**overloaded mental ledger**，过载的心理账本。你的大脑在强行维护一张多线程任务清单，每个任务的状态、进展、下一步动作全靠硬记。

这不是一个 UI 问题，这是一个认知负担问题。

## 一个视图，解决所有会话管理

Agent View 的设计极简——一个列表，所有会话。

运行 `claude agents`，或者在任何会话中按左方向键，就能进入这个统一视图。你看到的不是花哨的图形界面，而是纯终端里的一个列表。每一行显示：会话名称、当前状态（工作中/等待输入/已完成/失败/空闲/已停止）、最后一条回复的内容、最后交互的时间。

就这么一点改变，解决了一个大问题——**你不需要再记住任何东西了。看一眼就知道。**

关键操作只有几个：

- **Peek 预览**：选中一个会话，按空格键查看最后一轮对话内容
- **内联回复**：如果会话在等你决策，直接在预览里回复，不用进入全屏
- **全屏接入**：按回车进入完整会话，浏览全部记录
- **右方向键**：回到之前正在进行的会话

## 两个操作，改变工作流

Agent View 真正有意思的地方，是它引入了两个新的工作流操作：



**`/bg` —— 把当前会话扔到后台**

以前你想同时做多件事，得手动切标签页、开新终端、记位置。现在一句 `/bg`，当前会话进后台，回到 Agent View，发起新任务。所有状态自动维护，不需要你操心。

**`claude --bg` —— 从后台启动新任务**

直接以后台模式启动一个新 Agent 会话，不占用前台。搭配 Skill 使用效果更明显——批量提交几个任务，过一会回来审查结果。

这两个操作的组合，把 Claude Code 的使用方式从一个「对话式交互」变成了一个「异步协作式交互」。Agent 在后台独立工作，你只在需要决策时介入。

## 真实使用场景

早期用户已经给出了一些很具体的用法：

**1. 批量发起，统一收网**

写一个 Prompt 模板（或者 Skill），一口气启动五六个任务——每个处理不同文件、不同方向。过段时间回来，收获一批等待审查的 PR。这比「一个个来」的效率提升不是线性的，是倍数级的。

**2. 带状态监控的后台 Agent**

PR 看护者、仪表盘更新器、定时代码扫描——这些需要长时间运行的任务，Agent View 的列表会直接显示它们的下一次运行时间。你再也不用靠闹钟来提醒自己「去看看那个 Agent 跑完没」。

**3. 不打断主任务的快速查询**

一个很实际的场景：你在深度重构一个模块，突然想查一下某个函数的定义。以前你得中断当前会话、新建一个——等回来时思路断了。现在直接按左方向键进入 Agent View，发一个快速查询任务，Peek 看到结果，右方向键回到原会话。思路不断，上下文不丢。

**4. 扫一眼就知道交付状态**

多个会话跑完后，状态指示器加上 Peek 中的标题，快速扫描哪些会话生成了 PR、哪些还需要你介入。这个过程从「逐个检查」变成了「扫描仪表盘」。

## 拆开看：Agent View 的本质是什么

从产品角度看，Agent View 不是在加功能，是在**移除一个隐含的限制**——「一个终端一次只能安心做一个事」。

之前 Claude Code 的正确用法是「一个会话聚焦一个任务」。但当你真的有多个任务要并行时，工具本身没有提供管理多个会话的能力。用户被迫用自己的大脑来管理——这就是那个「心理账本」的来源。

这事情值得拉出来说清楚：**工具的责任边界在哪里？**

Claude Code 之前对「会话管理」的态度是：我只管当前会话的事，其他会话你自己管。这很合理——传统 CLI 工具都这样。但当 Agent 越来越强、跑的任务越来越长时间时，这个假设就不成立了。

Agent View 的核心洞察是：**当 Agent 能独立自主工作时，用户和 Agent 的关系就从「对话」变成了「协作」。** 协作就需要一个项目管理界面。

这也不是 Anthropic 的独创——IDE 里的任务列表、CI/CD 的管道视图、甚至是飞书的群聊列表——都是一个逻辑：当并发数量超过 3 个时，人类需要一张「一览表」来维持掌控感。

## 对独立开发者的意义

独立开发者通常是一个人干所有事。bug 要修、PR 要看、文档要写、测试要跑——精力本来就分散。Agent View 的价值在于：

1. **释放大脑缓存**。不用再记「那个会话在干嘛」，交给列表去管理
2. **降低启动成本**。以前开一个新任务的心理阻力来自于「开了之后怎么切回去」，现在不存在了
3. **批量思维**。一个补丁的心态从「做一件算一件」变成「启动一批，收获一批」

这三个变化加在一起，工作效率的提升不止是"快了一点"，而是工作方式本身变了。

## 说一个可能被忽略的点

Agent View 目前是 Research Preview（研究预览版），不是正式版。这意味着它的 API 和交互方式可能还会变。但方向已经很明显了——**Claude Code 在从「一个终端里的编码助手」变成「一个人工智能的操作系统」**。

这让我想起一个观点：好的工具不是让你跑得更快，是让你跑的方式不一样。

Agent View 就是这种「不一样」。它没有让单个 Agent 跑得更快，没有让模型更强，没有加任何新能力——但它让你的思维模式和工具的交互方式变了。从「一对一对话」到「一对多协作」，这个跃迁才是真的价值。
